## What This Task Does
Author the static portion of the ingest prompt that M24.T2's ingest runner will assemble and send to `claude -p`. The prompt instructs the synthesis worker to read the current wiki + new sources and produce updated entity pages plus exactly one log line â€” all within the wiki home path. The file ships with the wheel via the `prompts/*.md` package-data rule established in M23.T1.

## Key Architecture Constraints
- **Boundary:** prompt must explicitly state the worker writes only inside the wiki home. Bridge has already enforced this in code (safe_write); the prompt reinforces it so Claude's Edit/Write tool use stays inside the sandbox even when misdirected.
- **Tool allowlist:** prompt names only `Read`, `Edit`, `Write`. No `Bash`, `WebFetch`, or subprocess tools â€” synthesis is pure-text work.
- **Log contract:** prompt specifies the exact log line format (`YYYY-MM-DD | INGEST | <sources> â†’ <pages-touched>`) so the runner in M24.T2 can parse it reliably.
- **Examples drive behavior:** 2 good and 2 bad examples. Bad examples call out the common failure modes (inventing claims, silently dropping contradictions) so the worker is pre-loaded against them.
- **Snapshot-pinned:** any change to the template must appear in a test diff, not slip through silently. Guards against LLMs or humans editing the prompt without review.
- **Stdlib only:** loaded via `importlib.resources.files("claude_bridge.prompts") / "wiki_ingest.md"`.

## Design

### File: `src/claude_bridge/prompts/wiki_ingest.md`

The bundled prompt contains the **static** part of the ingest instruction. The M24.T2 runner assembles a full prompt by appending three runtime sections:

```
<static template: wiki_ingest.md>
---
## Current Schema
<contents of wiki/schema.md>
## Current Index
<contents of wiki/index.md>
## Existing Entity Pages
<listing of wiki/*.md with first 40 lines each>
## New Sources
<JSON-formatted SourceRecord[] from M23.T2 collect_sources>
---
Now synthesize. Begin by calling Read on any pages you need to update.
```

This task ships only the static part. The runtime assembly is M24.T2.

### Static template structure

- **Role preamble** â€” "You are a synthesis agent for the Bridge Wiki."
- **What this wiki is** â€” one paragraph, deferring details to the appended schema.
- **Tools available** â€” Read, Edit, Write (explicit allowlist; no Bash).
- **Boundary rule** â€” never write outside the wiki home; paths given to Edit/Write must be relative or under `~/.claude-bridge/wiki/`.
- **Output protocol** (numbered steps):
  1. Read existing pages referenced by new sources
  2. Update each touched page's Summary, Key Facts, Cross-references, Open Questions
  3. Create new pages with the template in `schema.md` when an entity has no home
  4. Update `index.md` if any new entity pages were created
  5. Append **exactly one** log line to `log.md` in the documented format
- **Log line format** â€” verbatim example with placeholders
- **Examples** â€” 2 good + 2 bad, each ~5 lines

### Loader helper

Refactor the existing `_load_schema_template()` into a general `_load_template(name)` in `wiki.py`. `_load_schema_template()` becomes a one-line wrapper that calls `_load_template("wiki_schema_template.md")`. This keeps M23.T1 tests passing and sets up clean reuse for M24.T2.

### Public surface for tests

Tests load the template via `importlib.resources` directly â€” no new public function needed. The `_load_template` refactor is implementation detail.
